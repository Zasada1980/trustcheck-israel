# **Полный технический и юридический анализ прямого доступа к государственным базам данных Израиля для B2C-приложений**

## **Аннотация**

В текущем цифровом ландшафте Израиля наблюдается фундаментальный сдвиг от закрытых, монолитных правительственных систем к архитектуре, ориентированной на API и открытые данные. Для разработчиков B2C-приложений и предпринимателей, стремящихся исключить посредников (таких как кредитные бюро, юридические агрегаторы и реселлеры данных), открываются новые возможности прямой интеграции. Однако этот ландшафт неоднороден: он представляет собой лоскутное одеяло из современных REST API, устаревших файловых архивов и закрытых юридических систем. Настоящий отчет представляет собой исчерпывающее исследование механизмов прямого доступа к четырем ключевым государственным ресурсам: Регистратору компаний (Rasham HaChavarot), Судебной системе (Net HaMishpat), Налоговому управлению (Rashut HaMisim) и Банку Израиля (Bank of Israel). Анализ охватывает технические протоколы (API, форматы файлов), юридические аспекты массового сбора данных (скрейпинг, защита конфиденциальности), структуру государственных сборов на 2025 год и требования к лицензированию.

**Глава 1\. Экосистема электронного правительства Израиля: Архитектура и Политика Открытых Данных**

### **1.1 Эволюция цифровой инфраструктуры: От "Tehila" до "Gov.il"**

Исторически цифровая инфраструктура правительства Израиля развивалась фрагментарно. Каждое министерство создавало свои собственные "информационные острова", что привело к разрозненности стандартов и протоколов доступа. Основой государственной сети долгое время служила инфраструктура "Tehila" (Правительственная интернет-инфраструктура), обеспечивающая безопасный хостинг и доступ. Однако в последнее десятилетие, под эгидой инициативы "Цифровой Израиль" (Digital Israel) и Управления ИКТ (Government ICT Authority), начался процесс консолидации ресурсов под доменом gov.il и порталом открытых данных data.gov.il.1

Для архитектора B2C-приложения понимание этой эволюции критически важно, так как приходится взаимодействовать с системами разных поколений. Если Налоговое управление (Shaam) активно внедряет современные стандарты OAuth2 и JSON REST API в рамках реформы цифровых счетов-фактур, то судебная система "Net HaMishpat" остается "крепостью", построенной на технологиях начала 2000-х годов, где доступ жестко контролируется физическими смарт-картами и VPN-туннелями. Портал api.data.gov (по аналогии с американским стандартом) декларирует стремление к открытости, но реальная глубина данных варьируется от министерства к министерству.2

### **1.2 Политика Open Data против Реальности**

Израильское правительство формально привержено принципам открытых данных, что подтверждается наличием централизованного портала данных, публикуемого Open Knowledge Foundation и правительством.1 Тем не менее, разработчики сталкиваются с дихотомией "Публичные данные" (Public Data) против "Ограниченных данных" (Restricted Data).

Публичные данные, такие как статистика, базовые реестры и тарифы, доступны для анонимного скачивания или через открытые API без аутентификации. Ограниченные данные, представляющие наибольшую ценность для B2C-сегмента (проверка кредитоспособности, налоговый статус конкретного дилера, детали судебных тяжб), требуют сложной процедуры идентификации, регистрации в качестве "Программного дома" (Beit Tochna) или получения специализированных лицензий. Прямой доступ к таким данным без посредников технически возможен, но требует преодоления значительных бюрократических и технических барьеров, которые мы подробно разберем в соответствующих разделах.

**Глава 2\. Регистратор Компаний (Rasham HaChavarot): Корпоративная Прозрачность**

Регистратор компаний, действующий в рамках Управления корпораций (Rishut HaTaagidim) Министерства юстиции, является "источником правды" о юридических лицах в Израиле. Для B2C-приложения, предлагающего услуги проверки контрагентов, работодателей или продавцов, эта база данных является фундаментальной.

### **2.1 Технические механизмы доступа**

#### **2.1.1 Официальные правительственные API**

В отличие от многих других ведомств, Регистратор компаний предоставляет относительно современный программный интерфейс, ориентированный на соблюдение процедур "Знай свой бизнес" (KYB \- Know Your Business) и противодействие отмыванию денег (AML).3 Этот API позволяет разработчикам получать статус юридических лиц в режиме реального времени, минуя дорогостоящих посредников.

Архитектура API и конечные точки:  
Доступ к данным осуществляется через шлюз data.gov.il, который часто использует архитектуру CKAN для управления наборами данных. Основные возможности интеграции включают:

* **Поиск и валидация:** API позволяет выполнять запросы по номеру компании (H.P. \- Het-Pey) или названию.  
* **Возвращаемые данные:** Ответы сервера (обычно в формате JSON) содержат критически важные поля:  
  * company\_id: Уникальный 9-значный идентификатор.  
  * company\_hebrew\_name / company\_english\_name: Официальные наименования на иврите и английском.4  
  * company\_status: Статус активности (например, "Active" (Peila), "Dissolved" (Mehuslet), "Violating" (Meferah)). Статус "Meferah" (нарушитель) является ключевым индикатором риска для пользователей B2C-приложения, указывающим на неуплату пошлин или несдачу отчетности.  
  * company\_exists: Булево значение для быстрой валидации форм.4  
  * corporation\_type: Тип корпорации (частная, публичная, государственная).

Интеграция может быть реализована как прямой вызов REST API или через механизмы WebHook, как это реализовано в платформах типа Cellosign, где изменение поля формы инициирует запрос к правительственному шлюзу для автозаполнения данных.4

#### **2.1.2 Массовые данные (Open Data Files)**

Для аналитических целей или создания локального кеша (что рекомендуется для снижения задержек и зависимости от аптайма правительственных серверов), Регистратор предоставляет возможность скачивания полных дампов базы данных.

* **Форматы:** Данные доступны в виде файлов с разделителями (CSV) или XML, а также в специфических форматах "Ascii structure", упакованных в ZIP-архивы.5  
* **Ограничения:** Публично доступные файлы часто содержат только "верхнеуровневую" информацию (название, адрес, статус). Глубокая информация о структуре акционеров (Shareholders), залогах (Shiabudim) и истории изменений часто исключена из бесплатных дампов и доступна только через платные выписки или специализированные запросы.

### **2.2 Структура расходов и государственные сборы (2025 год)**

Прямой доступ к данным позволяет существенно сэкономить по сравнению с использованием API посредников (таких как D\&B или BDI), однако он не является полностью бесплатным, особенно когда речь идет о юридически значимых документах.

#### **2.2.1 Ежегодные сборы (Agra Shnatit)**

Хотя эти сборы оплачиваются самими компаниями, информация о задолженности по ним является публичной и важной для оценки надежности бизнеса. На 2025 год установлены следующие тарифы, которые B2C-приложение может отображать пользователям как индикатор добросовестности компании:

* **Льготный тариф:** Для компаний, оплачивающих сбор до 31 марта 2025 года, сумма составляет **1 306 шекелей**.7  
* **Стандартный тариф:** С 1 апреля 2025 года сумма возрастает до **1 734 шекелей**.8  
* Приложения могут использовать API для проверки, оплатила ли компания сбор, так как неоплата автоматически переводит компанию в статус "Нарушитель" (Meferah), что влечет за собой ограничения на совершение юридических действий и возможность наложения личных санкций на директоров.

#### **2.2.2 Стоимость выписок (Nesach)**

Для получения полной юридической картины (например, кто является директором или есть ли залоги на имущество) требуется выписка (Nesach HaChavra).

* **Базовая информация:** Доступна бесплатно через веб\-сайт и API (валидация существования и статуса).10  
* **Полная выписка (Nesach Maleh):** Является платной услугой. Стоимость варьируется в зависимости от канала получения, обычно составляя от **11 до 36 шекелей** за выписку. При массовом автоматизированном получении через API для коммерческого использования могут применяться специальные тарифные соглашения с Управлением корпораций, однако для стартапа проще использовать модель "on-demand", перекладывая стоимость на пользователя или абсорбируя ее в рамках подписки.

### **2.3 Юридические аспекты и "Скрейпинг" корпоративных данных**

С точки зрения легальности массового сбора данных, корпоративный реестр является наименее рискованным активом.

* **Публичный характер:** Информация о юридических лицах по определению является публичной. Компании с ограниченной ответственностью (Ltd.) получают защиту акционеров в обмен на прозрачность.  
* **Персональные данные:** Важный нюанс заключается в данных о директорах и акционерах-физических лицах. Хотя эти данные находятся в открытом реестре, их массовая агрегация и профилирование (например, создание базы данных "Все компании, которыми владеет Иван Иванов") может подпадать под действие Закона о защите конфиденциальности (Privacy Protection Law), требуя регистрации такой базы данных в Управлении по защите конфиденциальности (PPA), так как она содержит информацию об "экономическом положении" физических лиц.6

**Вывод по главе:** Регистратор компаний предоставляет наиболее зрелый и доступный API для B2C-приложений. Рекомендуемая стратегия — использование бесплатного API для первичной валидации и статусов, и интеграция платного шлюза для получения полных выписок только по прямому запросу пользователя.

**Глава 3\. Судебная система (Net HaMishpat): Цифровая Крепость**

В отличие от Регистратора компаний, судебная система Израиля ("Net HaMishpat") представляет собой значительно более закрытую среду. Доступ к ней регулируется строгими процедурами идентификации, и на момент написания отчета (конец 2025 года) полноценный публичный API для поиска и скачивания судебных дел третьими лицами отсутствует.

### **3.1 Архитектура "Закрытого Сада"**

Система "Net HaMishpat" (Законная Сеть) разработана Управлением судов (Hanhalt Batei HaMishpat) в первую очередь как инструмент управления делами (Case Management System) для судей, секретариатов и сторон процесса, а не как публичный репозиторий.11

**Уровни доступа:**

1. **Внутренний контур:** Судьи и сотрудники аппарата.  
2. **Авторизованные внешние пользователи (Адвокаты):** Доступ осуществляется с использованием аппаратных смарт-карт (Smart Card) таких компаний, как Comsign или PersonalID.12 Это создает огромный барьер для автоматизации, так как требует физического подключения токена к серверу или использования сложных решений USB-over-IP.  
3. **Стороны процесса (Граждане):** Доступ через Национальную систему идентификации (Hizdahut Leumit). Позволяет просматривать только те дела, в которых гражданин является истцом или ответчиком.12  
4. **Публика:** Доступ крайне ограничен. Решения судов публикуются выборочно, и единого поискового API по всем текстам решений не существует.

### **3.2 Проблема отсутствия Open API и прецедент "Hashavim"**

Исследование показывает значительный пробел в прозрачности. В то время как в США существует система PACER (платная, но открытая для всех), в Израиле монополия на удобный поиск по судебным делам принадлежит частным агрегаторам (Takdin, Nevo), которые получают данные через исторически сложившиеся (и часто непубличные) соглашения или сложные методы сбора.

Прецедент "Hashavim":  
Попытки технологических компаний получить массовый доступ к данным через автоматизированный сбор (скрейпинг) столкнулись с жестким юридическим сопротивлением. В деле Hashavim HPS Business Information Ltd. v. Courts Administration рассматривалась законность "вытягивания" решений из базы данных. Хотя суд признал, что сами судебные решения являются публичными, он не обязал Управление судов предоставлять удобный интерфейс или API для коммерческих структур.14 Более того, условия использования "Net HaMishpat" прямо запрещают использование роботов и краулеров, что делает прямой скрейпинг юридически уязвимым (нарушение Закона о компьютерах в части несанкционированного доступа и нарушение контракта/ToS).15

### **3.3 Технические и юридические обходные пути для B2C**

Учитывая отсутствие API, B2C-приложение не может легально реализовать функцию "Найди все иски против этого сантехника" без участия самого сантехника. Однако существуют альтернативные архитектурные решения:

#### **3.3.1 Модель "Авторизованный Агент" (Authorized Agent)**

Вместо того чтобы приложение выступало как внешний скрейпер, оно может действовать от имени пользователя.

* **Механизм:** Пользователь (например, тот же сантехник, желающий доказать свою надежность) логинится в "Net HaMishpat" через интерфейс приложения, используя свои учетные данные Национальной системы идентификации.  
* **Интеграция:** Приложение использует сессию пользователя (Client-Side scraping или проксирование сессии) для извлечения справки об отсутствии открытых дел (Hieader Tviot).  
* **Легальность:** Этот метод юридически более чист, так как доступ осуществляется уполномоченным лицом (субъектом данных), а приложение выступает лишь инструментом.12

#### **3.3.2 Использование приложения "Tik Tech"**

Управление судов запустило приложение "Tik Tech" для граждан.12 Хотя официального API для интеграции с ним нет, анализ трафика этого мобильного приложения может выявить недокументированные JSON-эндпоинты, используемые для аутентификации и получения списка дел. Использование таких "скрытых" API является "серой зоной", но технически более надежно, чем парсинг HTML-страниц основного сайта.

### **3.4 Стоимость доступа**

* **Просмотр:** Для граждан просмотр своих дел бесплатен через Национальную систему идентификации.12  
* **Подача документов:** Сопряжена с уплатой судебных пошлин (Agrot), которые могут быть оплачены онлайн.  
* **Инфраструктура:** Для коммерческого доступа (если бы он был разрешен) потребовалась бы смарт-карта (стоимость ок. 400 шекелей в год) и, возможно, плата за массовую выгрузку, если такая опция будет введена в будущем по аналогии с Регистратором компаний.

**Вывод по главе:** Прямой массовый сбор данных из судов для B2C-приложения в настоящее время юридически опасен и технически сложен из\-за "Smart Card" барьера. Единственная жизнеспособная стратегия — модель, управляемая пользователем (User-Generated Access), где пользователь сам предоставляет доступ к своим данным через интерфейс приложения.

**Глава 4\. Налоговое Управление (Rashut HaMisim): Революция API**

Налоговое управление Израиля представляет собой наиболее прогрессивный фронт в области открытых данных, движимый не альтруизмом, а необходимостью борьбы с "черным рынком" и фиктивными счетами-фактурами. Реформа "Израильский счет-фактура" (Heshbonit Yisrael), вступившая в активную фазу в 2024-2025 годах, обязала бизнес работать через API, что открыло двери и для сторонних разработчиков.

### **4.1 Проект "Open API" и "Shaam"**

Техническим оператором систем Налогового управления является подразделение "Shaam" (Shirut Ibud Melacha \- Бюро обработки данных). Они развернули мощный шлюз API для поддержки реформы, требующей получения номера распределения (Mispar Haktza'a) для счетов-фактур свыше 25 000 шекелей в режиме реального времени.16

#### **4.1.1 Процесс регистрации "Программного дома" (Beit Tochna)**

Чтобы получить доступ к Production API, B2C-приложение должно пройти процедуру регистрации в качестве "Программного дома". Это ключевой этап легализации прямого доступа.  
Пошаговый алгоритм регистрации 18:

1. **Административная регистрация:** Необходимо зарегистрировать корпорацию в системе Shaam. Требуется назначить официального представителя.  
2. **Смарт-карта:** Администратор "Программного дома" обязан иметь личную смарт-карту (Comsign или PersonalID) для первоначального входа и подписания обязательств.  
3. **Портал разработчиков:** Регистрация на портале openapi-portal.taxes.gov.il. Здесь создается "Организация" и "Приложения" (Apps).  
4. **Юридические обязательства:** Подписание декларации об информационной безопасности и обязательств по неразглашению.  
5. **Вайтлистинг IP:** Shaam часто требует предоставления статических IP-адресов серверов, с которых будут идти запросы, для внесения в белый список брандмауэра.

### **4.2 Техническая архитектура и Аутентификация (OAuth2)**

Система использует стандартный протокол OAuth2, что делает интеграцию понятной для современных разработчиков.

Поток аутентификации (Authentication Flow) 18:

* **Типы токенов:**  
  * Long-Time Token: Токен с длительным сроком действия (месяцы), используемый для получения короткоживущих токенов доступа без повторного ввода пароля пользователем.  
  * Access Token: JWT-токен, действующий короткое время (например, 30 минут), который передается в заголовке Authorization: Bearer \<token\> каждого запроса.  
  * Refresh Token: Используется для обновления Access Token.  
* **Контекст пользователя:** API различает запросы "от имени приложения" (Application Scope) и "от имени пользователя" (User Scope). Для проверки статуса дилера часто достаточно прав приложения, тогда как для подачи отчетов требуется делегирование прав от конкретного налогоплательщика.

### **4.3 Ключевые API для B2C: Проверка статуса "Osek Murshe"**

Для B2C-приложения (например, маркетплейса услуг или аренды) критически важно проверить, является ли поставщик легальным бизнесом. Налоговая предоставляет для этого мощные инструменты, заменяющие устаревшие методы.

**Конечная точка: Проверка ведения книг (Ishur Nihul Sfarim)**

* **Функционал:** Позволяет проверить статус дилера (Osek Murshe) и ставку налога у источника (Nikuy Mas BaMakor).16  
* **Входные параметры:** Номер дилера (9 цифр).  
* **Выходные данные:**  
  * Статус (Активен/Закрыт/Приостановлен).  
  * Наименование бизнеса.  
  * Процент вычета налога (0% — идеально, выше — риск проблем с налоговой).  
* **Преимущества прямого доступа:** Использование этого API позволяет валидировать контрагентов в реальном времени абсолютно легально и бесплатно (после настройки интеграции), избегая платы за каждую проверку сторонним сервисам.  
* **Доступ без идентификации:** Некоторые сервисы (Open Access) доступны без токена пользователя, только с Client ID приложения, что идеально для массовых публичных проверок.20

**Вывод по главе:** Налоговое управление предоставляет самый современный и легальный путь для получения бизнес-данных. Регистрация в качестве "Программного дома" — это инвестиция времени (бюрократия), которая окупается отсутствием переменных расходов на посредников в будущем.

## ---

**Глава 5\. Финансовые данные и Банк Израиля**

Ландшафт финансовых данных в Израиле строго дихотомичен: "Негативные данные" (ограниченные счета) доступны относительно свободно в виде файлов, а "Позитивные данные" (кредитная история) находятся за высоким забором лицензирования.

### **5.1 Ограниченные счета (Hashbonot Mugbalim)**

База данных ограниченных счетов (клиентов, у которых вернулось 10 и более чеков без покрытия) является публичной согласно Закону о чеках без покрытия. Это критический ресурс для оценки риска в B2C.

**Механизм доступа:**

* **Отсутствие API:** Банк Израиля не предоставляет REST API для этого реестра. Данные публикуются в виде файлов на веб\-сайте.5  
* **Форматы файлов:**  
  * Файлы предоставляются в форматах "Ascii structure" (фиксированная ширина полей) или Excel, упакованные в ZIP-архивы.  
  * Разделение на "Корпорации" (Таагидим) и "Частные лица" (Ехидим). Для частных лиц доступ к массовым файлам ограничен и требует специального запроса с обоснованием цели (например, верификация чеков).5  
* **Техническая реализация:** B2C-приложение должно реализовать ETL-процесс (Extract, Transform, Load): ежедневный скрипт, который скачивает ZIP-файл, распаковывает его, парсит специфическую структуру (учитывая кодировку иврита, часто ISO-8859-8 или Windows-1255) и обновляет внутреннюю базу данных.  
* **Юридический риск:** Публикация устаревших данных может привести к иску о диффамации (Lashon Hara). Если ограничение со счета снято сегодня утром, а приложение показывает его вечером как ограниченный, это основание для иска. Обновление должно быть синхронным с публикацией файла БИ.

### **5.2 Система кредитных данных (Ma'agar Netunei Ashrai)**

Закон о кредитных данных (2016) создал центральный регистр, но прямой доступ к нему для стартапов практически закрыт.23

**Барьеры входа:**

1. **Лицензирование:** Доступ разрешен только лицензированным "Кредитным бюро" (Lishkat Ashrai).  
2. **Требования к капиталу:** Для получения лицензии требуется минимальный собственный капитал. Согласно нормативным актам, он варьируется от **500 000 до 4 000 000 шекелей** в зависимости от типа лицензии и объема деятельности.24  
3. **Безопасность:** Соответствие стандарту ISO 27001 и прохождение аудитов кибербезопасности.

Альтернатива: Open Banking (Открытый Банкинг)  
Вместо попыток получить лицензию Кредитного бюро, B2C-приложениям следует ориентироваться на Закон об услугах финансовой информации (2021).

* **Стандарт:** Основан на европейском стандарте Berlin Group NextGenPSD2.26  
* **Лицензия:** Лицензия "Провайдера финансовой информации" (Baal Rishayon LeMatan Sherut Meida Finansi) требует меньшего капитала (от **200 000 шекелей**) и позволяет получать данные о транзакциях напрямую из банков (Leumi, Hapoalim и др.) с согласия пользователя. Это де\-факто дает ту же аналитическую ценность, что и кредитный регистр, но через другой юридический канал.

**Глава 6\. Юридические рамки массового сбора данных**

Вопрос легальности "массового получения данных" (Mass Data Retrieval) является центральным для бизнес-модели, избегающей посредников. В Израиле это регулируется Законом о защите конфиденциальности (1981) и прецедентным правом.

### **6.1 Регистрация Баз Данных (Rishum Ma'agar Meida)**

Любое B2C-приложение, которое скачивает и хранит данные о физических лицах (даже из открытых источников, таких как судебные календари или списки ограниченных счетов), создает "Базу данных" в терминологии закона.

Обязательная регистрация требуется, если 27:

1. База содержит данные более чем о 10 000 человек.  
2. База содержит "чувствительную информацию" (информация об экономическом положении, долгах, здоровье).  
3. База используется для прямой рассылки (Direct Mailing).

**Импликация:** Приложение, агрегирующее данные о банкротствах или ограниченных счетах, **обязано** зарегистрировать свою базу в Управлении по защите конфиденциальности (PPA). Несоблюдение этого требования является уголовным преступлением (до 1 года тюрьмы).

### **6.2 Принцип "Целевого ограничения" (Purpose Limitation)**

Израильское право (и новые директивы PPA, в том числе касающиеся AI 28) требует, чтобы данные использовались только для той цели, для которой они были собраны или опубликованы.

* **Суды:** Данные публикуются для обеспечения гласности правосудия, а не для скоринга кредитоспособности. Массовый скрейпинг для создания "черных списков" жильцов или заемщиков может быть признан нарушением этого принципа.  
* **Решение:** Единственный безопасный путь — получение *явного согласия* пользователя на сбор данных о нем из государственных источников в момент регистрации в приложении.

### **7.2 Рекомендуемая гибридная архитектура**

Для B2C-приложения, стремящегося к максимальной независимости и легальности, рекомендуется следующая архитектура:

1. **Модуль "Бизнес-Валидатор":** Прямая интеграция с API Налогового управления (Shaam) для проверки статуса дилера и API Регистратора компаний для проверки юридического лица. Это закрывает потребность в проверке контрагентов (B2B2C).  
2. **Модуль "Финансовый Риск":** ETL-скрипт для ежедневной загрузки и парсинга файлов ограниченных счетов с сайта Банка Израиля. Данные хранятся во внутреннем Redis-кластере для быстрого поиска.  
3. **Модуль "Судебная История":** Отказ от массового скрейпинга. Реализация функции "Подключить мой аккаунт Net HaMishpat", где пользователь вводит свои данные, а приложение (на стороне клиента или через защищенную сессию) подтягивает *его* историю дел для отображения в профиле.  
4. **Модуль "Финансы Пользователя":** Получение лицензии Провайдера финансовой информации (Open Banking) для доступа к банковским транзакциям, вместо попыток пробиться в Кредитный регистр.

## **Заключение**

Прямой доступ к государственным базам данных Израиля в 2025 году — это решаемая инженерная и юридическая задача, но не тривиальная. Эпоха "дикого скрейпинга" ушла в прошлое из\-за ужесточения киберзащиты и законодательства о конфиденциальности. Будущее принадлежит авторизованным интеграциям через официальные шлюзы API (Налоговая, Регистратор) и регулируемому доступу к финансовым данным (Open Banking).

Наибольший потенциал для B2C-приложений кроется в использовании API Налогового управления, которые предоставляют бесплатные, легальные и точные данные о бизнесе в реальном времени, что ранее было прерогативой дорогих информационных брокеров. Однако для работы с данными судов и кредитной историей посредничество (либо лицензированное, либо через самого пользователя) остается неизбежным условием легального функционирования.
