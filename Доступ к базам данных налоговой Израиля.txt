Исследование механизмов легального доступа к фискальным базам данных Израиля: Реестр «Osek Murshe» и система «Ishur Nihul Sfarim»
Аннотация
В данном аналитическом отчете представлено исчерпывающее исследование правовых, технических и административных механизмов, регулирующих доступ к базам данных Налогового управления Израиля (Israel Tax Authority — ITA). Основной фокус исследования сосредоточен на двух ключевых запросах: легальное получение полного реестра авторизованных дилеров («Osek Murshe») и подключение к сервисам проверки статуса ведения бухгалтерских книг («Ishur Nihul Sfarim»). В условиях цифровой трансформации израильской экономики, ускоренной принятием Закона о хозяйственном регулировании (Economic Efficiency Law) на 2023–2024 годы, архитектура доступа к государственным данным претерпела фундаментальные изменения. Переход от статических файловых обменов к динамическим API-интерфейсам в рамках модели «Israel Invoice» создал новую парадигму взаимодействия бизнеса и государства.1
Отчет детально анализирует инфраструктуру службы компьютерной обработки «Шаам» (SHAAM), классифицирует методы интеграции (от системы массовой сверки «Mivzak» до современных протоколов Open API), рассматривает юридические риски неавторизованного сбора данных (скрейпинга) и предлагает пошаговые алгоритмы для регистрации в качестве «Дома программного обеспечения» (Software House). Исследование подтверждает, что, хотя полная выгрузка базы данных в формате Open Data недоступна для широкой публики по причинам защиты персональных данных, существуют строго регламентированные каналы, позволяющие верифицировать контрагентов в режиме реального времени и формировать локальные «базы знаний» для целей налогового комплаенса.

1. Введение: Экосистема фискальных данных Израиля
Цифровизация государственного управления в Израиле достигла уровня, при котором доступ к фискальным данным становится не просто вопросом прозрачности, а критическим условием операционной деятельности любого предприятия. Центральным узлом этой экосистемы является Налоговое управление Израиля и его технологическое подразделение — Служба автоматизированной обработки данных «Шаам» (Service Ibudim Memuchanim — SHAAM). Понимание архитектуры «Шаам» необходимо для любого субъекта, стремящегося легально взаимодействовать с реестрами налогоплательщиков.
1.1. Разграничение реестров: Корпоративный vs. Налоговый
Для корректного построения стратегии получения данных необходимо провести четкую демаркационную линию между двумя типами государственных реестров, которые часто смешиваются в сознании внешних наблюдателей:
1. Реестр компаний (Rasham HaChavarot): Находится в ведении Министерства юстиции (Управление корпораций). Он содержит информацию о юридических лицах (LTD/Baam), включая их регистрационные номера, адреса и статус (активна/ликвидирована).3 Данная информация является публичной, и выписки из реестра доступны любому желающему за небольшую плату или через открытые API Министерства юстиции. Однако этот реестр не содержит налоговой информации.
2. Реестр авторизованных дилеров (Osek Murshe): Находится в ведении Налогового управления (Министерство финансов). Этот реестр включает в себя не только компании, но и индивидуальных предпринимателей (Self-employed). Статус «Osek Murshe» — это фискальный атрибут, позволяющий взимать НДС. Именно этот реестр является предметом данного исследования.
Важность различия заключается в правовом статусе данных. Номер «Osek Murshe» для индивидуального предпринимателя совпадает с его номером удостоверения личности (Teudat Zehut). Следовательно, полная база данных всех дилеров содержит персональные данные миллионов граждан Израиля. Согласно Закону о защите конфиденциальности (Privacy Protection Law, 5741-1981), государство не имеет права публиковать эту базу в открытом доступе в виде единого скачиваемого файла (CSV/Excel), так как это нарушило бы права на приватность физических лиц-предпринимателей.5
1.2. Концепция «Базы знаний» в условиях ограничений
В свете вышесказанного, задача «получить всю базу знаний Osek Murshe» трансформируется из задачи однократного скачивания файла в задачу построения легального канала верификации. Налоговое управление не предоставляет «дамп» базы, но предоставляет инструменты для валидации любого конкретного идентификатора. Таким образом, «база знаний» на стороне клиента формируется накопительным итогом: бизнес запрашивает данные по своим контрагентам, получает официальные ответы от серверов «Шаам» и сохраняет их, формируя легальную, актуальную подборку данных, релевантную для его экономической деятельности.
Исторически существовали «серые» методы сбора этой базы, однако современные регуляторные требования, в частности внедрение модели «Israel Invoice» в 2024 году, делают прямую интеграцию с государственными API единственным устойчивым решением.1

	2.1. Система массовой сверки «Mivzak» (Mivzak System)
Для крупных организаций, которым необходимо поддерживать актуальность данных о тысячах контрагентов, основным инструментом на протяжении десятилетий являлась система «Mivzak». Это не API в современном понимании, а система файлового обмена.
Механизм работы:
Организация формирует текстовый файл определенного формата, содержащий идентификаторы (ID) всех своих поставщиков. Этот файл передается в «Шаам» через защищенный канал (или загружается через специализированный портал). В ответ система возвращает файл, обогащенный данными:
* Текущий статус «Ishur Nihul Sfarim» (действителен/недействителен).
* Процент налога, который необходимо удержать у источника (от 0% до максимальной ставки).
* Срок действия сертификата.8
Этот механизм де-факто позволяет крупным компаниям «выкачивать» актуальные данные по всему срезу рынка, с которым они взаимодействуют. Многие ERP-системы, такие как SAP и Oracle, имеют встроенные модули локализации для Израиля, которые автоматически генерируют и обрабатывают файлы формата «Mivzak».10
2.2. Система 1000 (Maarechet 1000)
Для компаний с меньшим объемом документооборота предназначена «Система 1000». Технически она схожа с «Mivzak», но имеет жесткое ограничение: в одном файле не может быть более 1000 записей. Если у компании 1001 поставщик, система либо отвергнет файл, либо потребует его разбиения.9 В последние годы Налоговое управление унифицирует форматы файлов для обеих систем, внедряя возможность идентификации по «номеру сущности» (например, номер компании), а не только по номеру налогового дела, что упрощает кросс-валидацию данных.13

3. Процедура получения «Ishur Nihul Sfarim»: От теории к практике
Сертификат ведения бухгалтерских книг («Ishur Nihul Sfarim») и справка об удержании налога у источника («Ishur Nikui Mas B'Makor») являются документами критической важности в израильском коммерческом обороте. Отсутствие валидного сертификата у поставщика обязывает плательщика удержать «максимальный налог» (обычно 45% или 50%) из суммы платежа и перечислить его государству. Это создает мощный экономический стимул для автоматизации проверки данного статуса.
3.1. Легальные способы подключения к базе данных сертификатов
Исследование показывает, что существует три легальных пути автоматизированного получения этой информации:
Путь А: Использование ERP-систем с сертифицированным интерфейсом
Для большинства корпоративных пользователей самым простым способом является использование готового программного обеспечения, которое уже прошло сертификацию в «Шаам».
* Пример: SAP Business One или Oracle Financials. Эти системы имеют встроенную функцию «Israel Localization». Пользователь (бухгалтер) должен лишь ввести учетные данные своей компании в настройках системы. ERP-система сама формирует файл запроса, отправляет его в «Шаам» и обновляет карточки поставщиков.10
* Преимущество: Не требуется разработка. Полная легальность.
* Недостаток: Требуется дорогая лицензия ERP.
Путь Б: Прямая интеграция через «Открытый API» (Open API)
Для финтех-компаний, разработчиков приложений и крупных холдингов, создающих собственные инструменты, единственным путем является прямая интеграция с API «Шаам».
Налоговое управление Израиля запустило портал разработчиков (openapi-portal.taxes.gov.il), который предоставляет доступ к современным RESTful сервисам.
Ключевые особенности Open API для сертификатов:
* Доступ осуществляется через протокол OAuth 2.0.
* Требуется строгая аутентификация с использованием смарт-карты (Smart Card).14
* Данные возвращаются в формате JSON.
* Сервис позволяет запрашивать данные как по конкретному дилеру, так и (в некоторых конфигурациях) получать обновления по списку авторизованных контрагентов.14
Путь В: Использование «Бюро данных» (Data Bureaus)
Если компания не хочет проходить сложную процедуру регистрации в «Шаам» (описанную ниже), она может легально купить доступ к данным у лицензированных агрегаторов, таких как Dun & Bradstreet Israel (D&B) или BDI Coface.
* Эти компании имеют оптовые соглашения с государством и агрегируют данные из Реестра компаний, Налогового управления и судебных реестров.
* Они предоставляют собственные API («Business Gateway»), которые возвращают те же данные (статус Osek Murshe, наличие сертификата), но через коммерческий шлюз.15 Это «белый» рынок перепродажи государственных данных.
3.2. Риски использования нелегальных методов (Скрейпинг)
Некоторые разработчики пытаются получить базу данных путем написания скриптов (ботов), которые перебирают номера дилеров на публичном сайте Налогового управления (www.misim.gov.il).
* Юридический аспект: Условия использования сайта (Terms of Use) обычно прямо запрещают автоматизированный сбор данных. Хотя израильская судебная практика в отношении скрейпинга публичных данных неоднозначна, нарушение условий использования государственного ресурса может повлечь блокировку IP-адресов и потенциальные гражданские иски.17
* Технический аспект: «Шаам» использует системы защиты (WAF), которые отслеживают аномальную активность и блокируют ботов. Построение бизнеса на скрейпинге гос. сайтов является неустойчивой стратегией.19

4. Революция «Israel Invoice» (Heshbonit Israel): Новый стандарт доступа к данным
С 2024 года ландшафт доступа к базе данных Osek Murshe радикально изменился с внедрением модели «Israel Invoice» (Heshbonit Israel). Согласно поправкам к Закону о хозяйственном регулировании, для вычета входящего НДС по сделкам свыше 25,000 шекелей (сумма будет снижаться ежегодно до 5,000 шекелей в 2028 году) требуется получение уникального «номера распределения» (Mispar Haktza'a) от Налогового управления в режиме реального времени.2
4.1. Как модель «Israel Invoice» открывает доступ к базе
Данная реформа фактически заставляет каждую B2B транзакцию проходить через серверы «Шаам».
* Для продавца: Чтобы получить номер распределения, продавец должен отправить данные счета (кто покупатель, сумма, НДС) через API. В ответ он получает подтверждение. Успешный ответ означает, что статус продавца активен и легален.
* Для покупателя: Покупатель получает API для верификации номера распределения. Отправляя номер счета и ID поставщика, покупатель получает подтверждение валидности.6
Это означает, что API «Israel Invoice» становится самым актуальным и точным источником правды о статусе «Osek Murshe». Любая компания, интегрированная в эту систему, автоматически получает доступ к механизму проверки контрагентов.
4.2. Техническая реализация доступа
API для модели «Israel Invoice» является частью общей платформы Open API.
* Используются форматы JSON.
* Обязательна цифровая подпись запросов.
* Система работает в режиме 24/7 с очень высоким SLA, так как от нее зависит возможность выписки счетов во всей стране.21
* В период «пилота» (2024 год) система присваивает номера всем запросам (кроме технических ошибок), но в будущем планируется внедрение алгоритмов выявления подозрительных транзакций на лету.1

5. Пошаговое руководство по легальному подключению (Open API)
Для технической реализации подключения к базе данных (как для «Ishur Nihul Sfarim», так и для «Israel Invoice») необходимо пройти строгую процедуру регистрации в качестве «Дома программного обеспечения» (Beit Tochna). Ниже представлен детальный алгоритм, основанный на официальной документации «Шаам».14
Этап 1: Административная регистрация
Прежде чем писать код, необходимо получить юридический статус субъекта, имеющего право на доступ к API.
1. Регистрация компании: Заявитель должен быть зарегистрированной в Израиле компанией или индивидуальным предпринимателем.
2. Подача заявки: Необходимо заполнить и отправить онлайн-форму (форма 130525 или аналогичная цифровая форма на портале Gov.il) для регистрации компании-разработчика ПО.22
3. Приложение по информационной безопасности: Критически важный шаг. Компания должна подписать «Приложение о безопасности данных» (Data Security Appendix). В этом документе компания обязуется:
   * Использовать стандарты шифрования (SSL/TLS).
   * Вести логи доступа.
   * Не передавать секретные ключи (Client Secret) третьим лицам.
   * Соблюдать требования Закона о защите конфиденциальности.22
Этап 2: Получение средств аутентификации
Доступ к порталу разработчиков невозможен по простой паре логин/пароль. Требуется «строгая аутентификация».
1. Смарт-карта (Smart Card): Представитель компании (Admin) должен приобрести аппаратную смарт-карту (токен) у одного из авторизованных удостоверяющих центров (Comsign или PersonalID). Карта должна быть выпущена на имя уполномоченного представителя компании.14
2. Регистрация в портале: С помощью смарт-карты производится первый вход на портал secapp.taxes.gov.il, где происходит привязка пользователя к компании-разработчику.
Этап 3: Техническая конфигурация на Портале Разработчиков
После административного одобрения (обычно приходит по email с номером заявки), открывается доступ к техническому порталу openapi-portal.taxes.gov.il.
1. Создание Организации: Admin создает профиль организации в портале.
2. Создание Приложения (App): В разделе «My Org Apps» создается новое приложение.
   * В этот момент генерируются Client_ID и Client_Secret.
   * Внимание: Client_Secret показывается только один раз. Его необходимо сохранить в защищенном хранилище (Vault).14
3. Определение Redirect URI: Необходимо указать URL, на который система будет возвращать пользователя после авторизации (Callback URL). Для локальной разработки допускается http://localhost.
4. Подписка на API Продукты: В каталоге API необходимо выбрать нужные продукты (например, «Israel Invoice Allocation», «Withholding Tax Info») и подписаться на них. Подписка может требовать дополнительного одобрения со стороны администраторов «Шаам».14
Этап 4: Реализация протокола OAuth 2.0
«Шаам» использует стандартный поток OAuth 2.0 Authorization Code Flow.
Шаг А: Получение кода авторизации
Приложение перенаправляет браузер пользователя на эндпоинт авторизации:
https://openapi.taxes.gov.il/shaam/tsandbox/longtimetoken/oauth2/authorize?response_type=code&client_id=&scope=&redirect_uri=
Пользователь (клиент вашего ПО) вводит свои учетные данные или использует свою смарт-карту на сайте Налогового управления. Это ключевой момент: доступ к данным делегируется самим владельцем данных. Программа действует от его имени.
Шаг Б: Обмен кода на токен
После успешного входа «Шаам» возвращает code на ваш Redirect URI. Сервер приложения должен сделать POST-запрос для получения токена:
* URL: .../oauth2/token
* Header: Authorization: Basic
* Body: grant_type=authorization_code&code=&redirect_uri=
В ответ приходит JSON с access_token (короткоживущий) и refresh_token (долгоживущий).
Шаг В: Использование токена
Для получения данных о «Ishur Nihul Sfarim» или для запроса номера счета, приложение отправляет запрос к API с заголовком:
Authorization: Bearer

6. Альтернативные источники и роль посредников
Учитывая высокую техническую и административную сложность прямой интеграции (необходимость смарт-карт, юридического лица в Израиле, прохождения проверок безопасности), на рынке сформировался сектор посредников.
6.1. Агрегаторы API (API Aggregators)
Компании, такие как «A-Point», «Linet», «Menahel4U», уже прошли все процедуры регистрации в качестве «Домов программного обеспечения». Они предоставляют упрощенные API для своих клиентов.25
* Схема работы: Клиент интегрируется не с «Шаам», а с API посредника. Посредник берет на себя сложности OAuth, подписи запросов и шифрования.
* Легальность: Это абсолютно легальная схема, так как посредник действует как уполномоченный представитель.
* Стоимость: Обычно взимается плата за транзакцию или месячная подписка (например, от 90 шекелей в месяц за пакет счетов).27
6.2. Кредитные Бюро (D&B, BDI)
Как упоминалось ранее, для задач анализа рынка или проверки контрагентов без проведения транзакций лучше подходят сервисы кредитных бюро.
* Они предоставляют данные не только о налоговом статусе, но и о кредитном рейтинге, судебных исках и структуре владения.
* Их API («Business Gateway») документированы на английском языке и ориентированы на международный бизнес.16

8. Комплаенс, безопасность и защита данных
Работа с базами данных «Шаам» накладывает серьезные обязательства в области информационной безопасности.
8.1. Суверенитет данных
Согласно инструкциям Налогового управления, данные, полученные через API, должны обрабатываться с соблюдением требований израильского законодательства. Хотя использование облачных провайдеров (AWS, Azure, Google Cloud) допускается, часто требуется, чтобы дата-центры находились в пределах определенных юрисдикций (обычно Израиль или ЕС, признанный «адекватным» по GDPR).30
8.2. Запрет на вторичное использование
Данные, полученные через интерфейс «Ishur Nihul Sfarim», предназначены исключительно для целей налогового удержания и верификации платежей. Использование этих данных для маркетинговых целей (например, выгрузка списка всех дилеров для спам-рассылки) является нарушением условий использования и Закона о защите конфиденциальности.
8.3. Аудит и мониторинг
«Шаам» ведет детальные логи всех запросов. Аномальная активность (например, запрос данных по 10,000 случайных ID за одну минуту) приведет к автоматической блокировке учетной записи разработчика. Легальный доступ подразумевает запросы только по тем контрагентам, с которыми у пользователя есть реальные или планируемые деловые отношения.

9. Заключение
Исследование показало, что получение «всей базы знаний Osek Murshe» в виде единого файла невозможно легальным путем из-за ограничений приватности. Однако, государство Израиль создало мощную, современную и полностью легальную инфраструктуру для динамического доступа к этой базе знаний.
Для бизнеса и разработчиков существует три основных стратегии:
1. Для корпоративного использования (Internal Use): Использование системы «Mivzak» через существующие ERP-системы для массовой сверки поставщиков.
2. Для создания продуктов (Build): Регистрация в качестве «Software House» и интеграция с Open API «Шаам» через протокол OAuth 2.0. Это путь для тех, кто строит финтех-решения, маркетплейсы или бухгалтерский софт.
3. Для быстрого старта (Buy): Использование API посредников (A-Point, D&B), которые берут на себя сложность взаимодействия с государством за комиссию.
Внедрение модели «Israel Invoice» в 2024 году делает интеграцию с API «Шаам» неизбежной для любого бизнеса, желающего легально функционировать на израильском рынке, фактически превращая API Налогового управления в центральный нервный узел всей B2B экономики страны.

Приложение: Технические спецификации и коды ответов
Для технических специалистов, планирующих интеграцию, ниже приведены критически важные технические детали, собранные в ходе исследования материалов разработчика.14
A.1. Коды ошибок API (HTTP Status Codes)
Понимание кодов ответов критично для отладки интеграции с «Шаам»:
* 200 OK: Запрос успешен, данные возвращены в теле JSON.
* 400 Bad Request: Ошибка в структуре запроса (неверный формат JSON, отсутствует обязательное поле, неверная контрольная сумма ID).
* 401 Unauthorized: Токен доступа истек или неверен. Решение: использовать refresh_token для получения новой пары ключей.
* 403 Forbidden: У клиента нет прав на запрашиваемый ресурс (Scope). Часто возникает, если приложение пытается запросить данные, на которые не подписано в портале.
* 404 Not Found: Указанный номер дилера (Osek Murshe) не найден в базе данных. Это может означать, что бизнес закрыт или номер введен с ошибкой.
* 422 Unprocessable Entity: Ошибка бизнес-логики (например, дата счета вне допустимого диапазона для модели «Israel Invoice»).
* 500 Internal Server Error: Внутренняя ошибка серверов «Шаам». Рекомендуется повторить запрос с экспоненциальной задержкой (Exponential Backoff).
A.2. Пример структуры запроса на получение токена (cURL)


Bash




curl -X POST \
 https://openapi.taxes.gov.il/shaam/tsandbox/longtimetoken/oauth2/token \
 -H 'Authorization: Basic' \
 -H 'Content-Type: application/x-www-form-urlencoded' \
 -d 'grant_type=authorization_code' \
 -d 'code=' \
 -d 'redirect_uri='

Примечание: В среде Production URL будет отличаться (без tsandbox).
A.3. Валидация номера «Osek Murshe»
Перед отправкой запроса к API рекомендуется проводить локальную валидацию номера. Для компаний (LTD) и ИП в Израиле используется алгоритм Луна (Luhn algorithm) или специфический алгоритм проверки контрольной суммы израильского ID (Teudat Zehut). Отсеивание некорректных номеров на стороне клиента снижает нагрузку на API и риск блокировки за подозрительную активность.
